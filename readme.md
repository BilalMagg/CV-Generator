# CV Generator

A full-stack, AI-powered platform for generating tailored CVs, tracking job applications, and managing your career portfolio. Built with microservices architecture, event-driven communication, and an intelligent AI agent pipeline.

## Features

- **AI CV Generation** — submit a job description and get a tailored CV generated by a pipeline of AI agents (job extraction, profile matching, template rendering, ATS optimization, email delivery)
- **Job Application Tracking** — manage applications with dashboard, kanban board, calendar, and analytics views
- **Portfolio Management** — manage projects, skills, experience, education, certifications, languages, social links, and more
- **Job Offer Aggregation** — live job crawling with real-time notifications via Kafka + SignalR
- **Email Notifications & Reminders** — scheduled reminders for application follow-ups, configurable notification preferences
- **Keycloak SSO** — centralized authentication and user management
- **Real-time Updates** — Kafka event-driven architecture for async communication between services

## Local Development (without Docker)

Docker is required for infrastructure (DBs, Kafka, Keycloak) and the AI agents. Backend services and frontend can run natively via `dotnet run` / `ng serve`.

### 1. Start infrastructure

```bash
# Start databases
docker compose up -d user-db content-db workflow-db application-db \
  job-offer-db notification-db cv-db keycloak-db

# Start Kafka + supporting services
docker compose up -d kafka schema-registry kafka-ui minio

# Start Keycloak (comes with keycloak-db)
docker compose up -d keycloak
```

> The first Keycloak startup imports the `cv-realm` realm from `api_gateway/keycloak-realm/`. If you need to reset Keycloak: `docker compose down -v keycloak keycloak-db && docker compose up -d keycloak keycloak-db`.

### 2. Start backend services

Each service reads its connection string from `appsettings.json` (which points to `localhost:5433–5439` for DBs). Kafka and gRPC URLs are configured via launch profiles.

```bash
# Terminal 1: User Service (REST :5001 · gRPC :50001)
cd backend/src/user-service && dotnet run

# Terminal 2: User Content Service (:5002)
cd backend/src/user-content-service && dotnet run

# Terminal 3: Workflow Service (:5003)
cd backend/src/workflow-service && dotnet run

# Terminal 4: Application Service (:5004)
cd backend/src/application-service && dotnet run

# Terminal 5: CV Service (:5008)
cd backend/src/cv-service && dotnet run

# Terminal 6: Job Offer Service (:5006)
cd backend/src/job-offer-service && dotnet run

# Terminal 7: Notification Service (:5007)
cd backend/src/notification-service && dotnet run

# Terminal 8: API Gateway (:8080)
cd api_gateway && dotnet run
```

> Port differences between Docker and local: Docker uses `8082–8088`, local dev uses `5001–5008` (with gRPC on `50001`). The API gateway's `appsettings.Development.json` overrides cluster destinations to localhost ports.

### 3. Start frontend

```bash
cd frontend
npm install
ng serve
```

Frontend dev server runs on `http://localhost:4200` and proxies `/api/*` to `http://localhost:8080` (api-gateway).

### 4. Start AI agents

```bash
docker compose up -d job-extractor search-agent template-agent \
  cv-optimizer contact-agent job-crawler
```

## Authentication Flow

```
Browser                         API Gateway                      Keycloak
   │                                │                                │
   │  GET /api/auth/login           │                                │
   │───────────────────────────────▶│                                │
   │  302 Redirect to Keycloak      │                                │
   │◀───────────────────────────────│                                │
   │                                │                                │
   │  Authenticate                  │                                │
   │───────────────────────────────────────────────────────────────▶│
   │  Auth Code                     │                                │
   │◀───────────────────────────────────────────────────────────────│
   │                                │                                │
   │  GET /api/auth/callback?code=  │                                │
   │───────────────────────────────▶│                                │
   │                                │  Exchange code for tokens      │
   │                                │───────────────────────────────▶│
   │                                │  Access + ID + Refresh tokens  │
   │                                │◀───────────────────────────────│
   │                                │                                │
   │                                │  POST /api/users/sync          │
   │                                │────────── user-service ───────▶│
   │                                │  X-User-Id returned            │
   │                                │◀───────────────────────────────│
   │                                │                                │
   │  Set cv_session cookie         │                                │
   │◀───────────────────────────────│                                │
   │                                │                                │
   │  GET /api/applications         │                                │
   │  (cv_session cookie)           │                                │
   │───────────────────────────────▶│                                │
   │                                │  Validate session,             │
   │                                │  attach Authorization header   │
   │                                │  + X-User-Id to downstream     │
   │                                │────────── backend ────────────▶│
```

- The gateway uses **cookie-based sessions** (cv_session) backed by in-memory auth state
- Downstream services trust the gateway and read `X-User-Id` header (or JWT as fallback)
- When the gateway restarts, in-memory sessions are lost and users must re-authenticate
- Registration (`/api/auth/register`) creates the user in Keycloak via Admin API and emits a `user.registered` Kafka event

## CV Generation Pipeline

The core workflow transforms a job description into a tailored CV:

```
1. User submits job description (URL or text)
          │
          ▼
2. Workflow Service creates workflow record
          │
          ▼
3. Job Extractor extracts structured requirements
   (role, required skills, experience level, keywords)
          │
          ▼
4. Search Agent matches user profile against requirements
   (RAG vector search on skills, projects, experience)
   → returns matched items + gap analysis + match score
          │
          ▼
5. Template Agent selects and fills CV template
   (Jinja2/LaTeX rendering)
          │
          ▼
6. CV Optimizer validates ATS compliance
   (keyword placement, formatting, section scoring)
          │
          ▼
7. Contact Agent delivers CV via email
   (SMTP/Gmail API with cover letter)
```

Endpoints:
- `POST /api/workflows/generate-cv` — trigger the pipeline
- `GET /api/workflows/{id}` — check status and results
